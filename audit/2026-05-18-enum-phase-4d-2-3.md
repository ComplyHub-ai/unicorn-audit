## Findings

### Why 4D-2 and 4D-3 were merged

The original plan had 4D-2 (column type change) and 4D-3 (RLS policy rewrites) as separate migrations. Two failed attempts revealed the blocker: PostgreSQL holds an AccessExclusiveLock on `public.users` during ALTER COLUMN TYPE and requires all 87 RLS policies referencing the column to be dropped first — even policies that reference the column by name without a type cast. Because the policy DROP and RECREATE must bracket the column change, both phases had to run in a single transaction.

Key learnings from the retry sequence:

- **Retry 1** (first attempt) — blocked by `trg_set_user_type_from_role` trigger with `UPDATE OF unicorn_role` column-level specificity. Fix: drop/recreate trigger around the column change.
- **Retry 2** — blocked by 87 RLS policies as dependents of the enum type. Fix: merge 4D-3 into same migration.
- **Retry 3** (final) — policy capture loop also captured `users_update_own`, which was being handled manually; would have created the policy twice. Fix: exclude `users_update_own` from the capture temp table (`AND NOT (schemaname='public' AND tablename='users' AND policyname='users_update_own')`), count becomes 86 in the loop + 1 manual recreate.
