## Decisions

- **Initial acting user only** — `acting_user_id` set at session start, never updated mid-session. Mid-session switch tracking (child table) deferred as a future enhancement.
- **FK target: `auth.users(id)`** — consistent with existing `actor_user_id` FK on the same table.
- **ON DELETE: `SET NULL`** — preserves audit row when a tenant user is deleted (correct for a witness column; `actor_user_id` uses CASCADE by contrast).
- **No index** — deferred until a query path needs it.
- **No backfill** — 433 legacy rows remain NULL.

## Open questions parked

- `actor_user_id` uses `ON DELETE CASCADE` while `acting_user_id` uses `ON DELETE SET NULL` — asymmetry is intentional (actor deletion should remove the whole row; acting-user deletion should preserve the audit record) but worth documenting in schema conventions.
