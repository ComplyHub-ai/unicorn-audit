## What shipped

### Feature ‚Äî Governance Documents page overhaul (`/client/governance-documents`)

Client-facing page that lists all compliance documents generated by Vivacity for each RTO. Multiple iterative Lovable deploys over one session.

---

**Deploy 1 ‚Äî Framework filter + Packages column (database view)**

Three changes shipped together:

1. **Framework filter** ‚Äî Previously the Framework dropdown and column used a hardcoded label string comparison. Changed to use `dd_governance_framework` (the same lookup table used on the admin `/manage-documents` side), filtering by `value` not label. This makes the filter consistent with admin behaviour and future-proof if labels change.

2. **Packages column ‚Äî active only, grouped in view** ‚Äî The Packages column previously ran 4 sequential JS queries and could show paused packages. Refactored to use a new database view `v_client_governance_documents` (`security_invoker = true`) with a single `STRING_AGG(DISTINCT p.name, ', ' ORDER BY p.name)` grouping active packages only (`membership_state = 'active'`). Carl's direction: "group should be in view and not joined." The view joins `document_instances ‚Üí documents ‚Üí stage_instances ‚Üí package_instances (active only) ‚Üí packages`.

3. **View button (renamed from Download)** ‚Äî The "Download" action button was renamed to "View" with an Eye icon. Changed to open the document in read-only mode by replacing `action=default` with `action=view` in the SharePoint URL. Prevents clients from accidentally opening documents in edit mode.

Migration: `20260529011832` ‚Äî `CREATE OR REPLACE VIEW public.v_client_governance_documents WITH (security_invoker = true)` with `GRANT SELECT` to `authenticated` and `service_role`.

---

**Deploy 2 ‚Äî SharePoint button (per-document folder navigation)**

Added a per-row "SharePoint" button (ExternalLink icon) that navigates to the specific SharePoint folder containing that document, not the base tenant governance folder.

Implementation:
- New edge function `get-sharepoint-parent-folder` ‚Äî takes `{ file_url, tenant_id }`, resolves the SharePoint URL to a `driveId`/`itemId` via `resolveDriveItemFromSharingUrl`, fetches the item's `parentReference.id`, then fetches that parent folder's `webUrl`
- Frontend: `handleOpenSharePointFolder` async handler with per-row loading state (`openingSharePointId`), spinner while loading, toast on failure
- Auth: service-role Supabase client + `getUser(token)` pattern (matching `verify-compliance-folder`)
- Zod validation: `{ file_url: string().url(), tenant_id: number().int().positive() }`

---

**Deploy 3 ‚Äî Framework column short labels**

Changed the Framework table column to display the short `value` ("CRICOS", "GTO", "RTO") instead of the full label ("CRICOS National Code", "GTO Compliance", "RTO Compliance"). Matches admin side display. Filter dropdown retains full labels for readability.

---

**Deploy 4 ‚Äî Scroll-to-top button**

Added a floating scroll-to-top button (fixed bottom-right, ChevronUp icon, outline Button variant) that appears when `window.scrollY > 300`. Smooth scrolls to top on click with opacity fade transition. Quality-of-life improvement for the document list.

### Bug fixes ‚Äî `get-sharepoint-parent-folder` (3 rounds)

The new edge function required three fix cycles before working correctly.

**Fix 1 ‚Äî Broken auth (`getClaims`):**
Initial deploy used `supabase.auth.getClaims(token)` which does not exist in `@supabase/supabase-js@2`. It silently returned null data, and because the Supabase client also used `SUPABASE_ANON_KEY`, the subsequent `tenant_sharepoint_settings` RLS query saw no authenticated user and returned null ‚Üí function returned 404 for all tenants. Fixed by switching to service-role client + `supabase.auth.getUser(token)` (same pattern as `verify-compliance-folder`).

**Fix 2 ‚Äî Wrong tenant gate:**
After the auth fix, six active tenants that have no row in `tenant_sharepoint_settings` still received 404. The existence check was the wrong guard ‚Äî the function uses app-level Graph credentials and doesn't depend on tenant-specific SharePoint settings. The `file_url` passed by the caller originates from an RLS-secured view, which is the real access boundary. Removed the `tenant_sharepoint_settings` check entirely. Made `tenant_id` optional in the Zod schema (frontend continues sending it, function ignores it). Auth check via `getUser` is retained as the sole auth boundary.

**Fix 3 ‚Äî Invalid Graph endpoint (`/parent`):**
After Fix 2, all tenants were still getting 404. Root cause: the function called `/drives/{driveId}/items/{itemId}/parent`, which is not a valid navigation property on DriveItem in Microsoft Graph v1.0. Graph returned 404, and the function propagated the Graph status code directly to the client. Fixed with a two-step lookup: (1) fetch the item with `$select=parentReference` to get `parentReference.id`; (2) fetch that parent item with `$select=id,name,webUrl`. This is the documented Graph pattern for retrieving a drive item's parent folder.

## Key decisions made this session

- **`v_client_governance_documents` view with `SECURITY INVOKER`** ‚Äî mandatory flag to prevent RLS bypass. Without it, the view runs as the definer and all tenants could see each other's documents.
- **`membership_state = 'active'` filter in view, not frontend** ‚Äî Carl's instruction to group in the view. Paused packages (`membership_state = 'paused'`) are excluded; only active packages appear in the Packages column.
- **Only show `status = 'generated'` documents** ‚Äî documents without a generated file are excluded from the client view. The query applies `.eq("status", "generated")` in the frontend.
- **`resolveDriveItemFromSharingUrl` used for all document URLs** ‚Äî `generated_file_url` is the `_layouts/15/Doc.aspx?sourcedoc=GUID` format. This was chosen over `generated_item_id` because only 201 of 23,598 generated documents have `generated_item_id` populated, whereas all generated documents have `generated_file_url`.
- **`tenant_sharepoint_settings` check removed from `get-sharepoint-parent-folder`** ‚Äî the check was designed for isolation but is the wrong boundary. The correct boundaries are: (a) `getUser(token)` confirms the caller is authenticated, (b) the `file_url` comes from an RLS-secured view, (c) Graph app credentials enforce SharePoint access.
- **Framework column shows value, not label** ‚Äî matches admin side; short values (CRICOS, GTO, RTO) are clearer in a narrow table column than full labels.
