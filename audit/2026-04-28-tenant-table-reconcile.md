## Finding 1 ŌĆö tenant_users vs tenant_members

### What was verified

Grepped `supabase/migrations/` for `CREATE TABLE` on both names.

| Table | Migration | Created |
|---|---|---|
| `tenant_users` | `20260105014522_c020bb23-ŌĆ”sql` | Initial schema batch |
| `tenant_members` | `20260105022505_3f033c5a-ŌĆ”sql` | RBAC Model Implementation migration |

Both tables exist and are in active use. Not a rename in flight.

### Schema comparison

**`tenant_users`:**
- `id bigint GENERATED BY DEFAULT AS IDENTITY`
- `tenant_id bigint NOT NULL REFERENCES tenants(id)`
- `user_id uuid NOT NULL`
- `role text NOT NULL DEFAULT 'child' CHECK (role IN ('parent', 'child'))`
- `primary_contact boolean` (added later, auto-set to `true` by `sync_primary_contact_on_role` trigger when `role = 'parent'`)
- `secondary_contact boolean` (added in migration `20260420035212_ŌĆ”`)

**`tenant_members`:**
- `id uuid PRIMARY KEY DEFAULT gen_random_uuid()`
- `tenant_id bigint NOT NULL REFERENCES tenants(id)`
- `user_id uuid NOT NULL`
- `role text NOT NULL DEFAULT 'General User' CHECK (role IN ('Admin', 'General User'))`
- `status text NOT NULL DEFAULT 'active' CHECK (status IN ('active', 'inactive', 'pending'))`
- `invited_at timestamptz`, `joined_at timestamptz`, `created_at`, `updated_at`
- Backfilled from `users.tenant_id` at migration time

### Usage split (grepped src/ and supabase/functions/)

`tenant_users` used in: client management UI components (`TenantUsersTab`, `ClientDetail`, `ManageTenants`, `PackageDetail`), email delivery (`send-email-graph`, `send-composed-email`, `process-notification-outbox`), document generation (`generate-document`, `generate-excel-document`, `bulk-generate-phase-documents`), invite-user skip-email path, M365 provisioning (`provision-m365-user`), academy enrolments (`useAcademyEnrollments`, `useAcademyPackageRules`), auth helpers (`_shared/auth-helpers.ts`), addin auth exchange.

`tenant_members` used in: auth (`useAuth`), billing signals (`useBillingSignals`), seat limits (`useSeatLimits`), AI features (`AskVivPanel`, `compliance-assistant`, `vector-search`, `strategic-orchestration`, `risk-command-engine`), TGA integration (`tga-integration`).

`useClientActingUser.ts` queries `tenant_users` first (primary_contact), then falls back to `tenant_members` (Admin role) ŌĆö confirming the two tables are complementary.

### Conclusion

Two distinct tables, both intentional and active. KB was incomplete ŌĆö only `tenant_members` was documented. Codebase wins.

### What was changed

- `pinned/glossary.md`: added table rows for `tenant_members` and `tenant_users` in Product-internal terminology
- `pinned/conventions.md`: expanded Identity shape section to document both tables with role values, key columns, and a two-table note
- `codebase-state/architecture.md`: added `tenant_users` to system overview block; added Two membership tables comparison table in Multi-tenancy section
