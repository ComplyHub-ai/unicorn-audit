## Findings

### BUG-033 — Test fixture uses invalid `eos_function_type` value (resolved)

**Surfaced:** Phase 5B audit (same session, earlier that day). `src/test/fixtures/eos-test-data.ts:263` set `function_type: "integrator"` on a mock `accountability_functions` row. `"integrator"` is an `eos_seat_role_type` value, not an `eos_function_type` value. The companion test at `accountability.test.tsx:54–59` found the fixture by that wrong value and asserted on it — the test passed only because the fixture itself was wrong. Once the Phase 5B FK constraint on `accountability_functions.function_type` lands and tests run against a real DB, this would have thrown a FK violation.

**Fix delivered (commit `ec1e426d`):**
- `eos-test-data.ts:263` — `function_type: "integrator"` → `"operations"`, `name: "Integrator"` → `"Operations"`.
- `accountability.test.tsx:54–59` — test description updated to "should include Operations function"; finder updated to `f.function_type === "operations"`; assertion updated to `expect(operations?.name).toBe("Operations")`.
- No production code touched. No DB changes.

**Note:** Lovable also changed `func-002` (the second fixture row) from `name: "Finance"` to `name: "Operations"`, making both fixture rows identical. Harmless for test purposes; no distinct function types are needed in this fixture today. Flag if a test ever needs two distinct `function_type` values in this fixture.

### BUG-040 — Dead `WHERE status = 'Done'` condition in `v_client_eos_summary` (resolved)

**Surfaced:** Phase 5A audit (earlier session, same day). The `completed_todos` subquery in `public.v_client_eos_summary` compared `et.status::text = 'Done'::text`. `'Done'` has never been a valid `eos_todo_status` value — valid values are `Open`, `Complete`, `Cancelled`. The condition always evaluated false; `completed_todos` returned 0 for every tenant since the view was created. Phase 5A preserved the dead condition byte-identical in the mandatory DROP + RECREATE (the type-change migration required dropping the view because `pg_depend` records a dependency on the enum type). BUG-040 was tracked and fixed independently.

**Fix delivered (migration `20260519030910_2a6fd031-f866-46ef-835c-6938e0396860.sql`, commit `ec1e426d`):**
- `CREATE OR REPLACE VIEW public.v_client_eos_summary` — `et.status::text = 'Done'::text` replaced with `et.status = 'Complete'` in the `completed_todos` subquery.
- All other subqueries, column names, aliases, and view structure preserved byte-identical.

**Pre-fix blast-radius check:**
- No component or hook queries `v_client_eos_summary` directly — all `src/` references are in the auto-generated `types.ts` (foreign-key metadata only).
- No dependent views, no RLS policies, no triggers on this view.
- `completed_todos` will now return real non-zero counts. Because no app code reads the column today, no UI will change at deploy time.

**Post-deploy verification:** Live DB queried via Supabase MCP. `pg_get_viewdef` confirms `et.status = 'Complete'::text` in `completed_todos` subquery. No `'Done'` string anywhere in the view definition.
